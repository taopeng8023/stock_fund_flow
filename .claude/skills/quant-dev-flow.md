# A股量化系统开发流程

## 触发条件

当用户在**本项目**中提出功能需求、优化需求、bug 修复时，自动启用本流程。

**不触发**：闲聊、知识问答、非本项目相关的话题。

---

## 核心原则

### 不要一直让用户确认

用户提出需求后，直接按流程推进。不要：
- ❌ "要我帮你做吗？"
- ❌ "你想用方案A还是方案B？"（有明显更优方案时直接选）
- ❌ "需要我继续吗？"
- ❌ "要不要我优化？"（用户说"优化"就是要做）

只有在以下情况才询问用户：
- 两个方案优劣完全相当，无法判断
- 需要用户提供数据（如 webhook URL、持仓信息）
- 改动会影响核心架构，风险极高

### 每个 commit 只做一件事

```
✅ 好的 commit：feat(risk): 黑天鹅阈值校准
✅ 好的 commit：fix(score): 排名跨日变化类型安全
❌ 坏的 commit：feat + fix + refactor 混在一起
```

### 亲自验证

写完代码后必须运行测试，不等用户说"测试一下"。测试失败就修，修到通过为止。

---

## 开发流程

### 阶段 1：需求分析 — 寻找最优解（15%）

这个阶段的目标不是"找到一个能用的方案"，而是**找到当前约束下的最优解**。

#### 1.1 深度理解需求

- 用户这句话背后想解决什么**实际问题**？
- 这个问题的**边界**在哪——影响哪些模块？不碰哪些模块？
- 有没有隐含的约束（性能、数据可用性、时间成本）？

#### 1.2 系统性探索现状

不是只读一个文件，而是画出影响范围：

- **直接相关**：需要改/新建哪几个文件？
- **间接影响**：改了之后会影响到哪些下游模块？
- **已有轮子**：现有代码里有没有可以复用的函数/模式/数据？

#### 1.3 候选方案对比（脑子里的权衡表）

至少过 2 个候选方案，比较：

| 维度 | 方案 A | 方案 B |
|------|------|------|
| 实现复杂度 | | |
| 是否新增依赖 | | |
| 对现有代码的侵入性 | | |
| 数据准确性/可靠性 | | |
| 后续维护成本 | | |

**决策原则**：
- 零依赖 > 新依赖（能用已有数据的就不调新 API）
- 融入架构 > 打补丁（新增代码按现有模式写，不另起炉灶）
- 数据驱动 > 拍脑袋（阈值有回测数据支撑，不凭空编数字）
- 简单 > 复杂（两个方案效果差不多时，选简单的）

#### 1.4 确定方案、开始动手

脑子里的权衡做完后，**直接开始写代码**，不写设计文档，不等待确认。除非两个方案在所有维度上完全打平，才问用户。

### 阶段 2：开发（70%）

1. **先看后写**：读要修改的文件，理解代码风格和模式
2. **跟着模式走**：新代码匹配现有代码的命名、注释密度、结构
3. **增量修改**：改最小范围，不碰无关代码
4. **注册/集成**：新模块记得注册到 `__init__.py`、`config.py`、`engine.py` 等入口

### 阶段 3：测试验证（15%）

1. **语法检查**：确保 Python 能 import，没有语法错误
2. **功能测试**：用实际数据跑一遍，看输出是否符合预期
3. **边界测试**：想一下极端情况（空数据、异常日期、缺失字段）
4. **集成测试**：如果改了公共接口，确保上下游模块不炸

### 阶段 4：修复 Bug（5%）

1. **看报错**：仔细读 traceback，定位根因
2. **修根因**：不改表面症状
3. **回归测试**：修完再跑一次，确保没引入新问题

### 阶段 5：提交（5%）

1. **commit 格式**：`类型(scope): 描述`
   - 类型：feat / fix / refactor / perf / docs
   - scope：模块名（risk / score / portfolio / notify / dashboard）
   - 描述：中文，简洁准确
2. **push**：commit 完直接 push，不等用户说

## 项目架构速查

```
portfolio/          ← 决策层（新建模块优先放这里）
  black_swan.py     ← 黑天鹅检测（13条规则）
  manager.py        ← 持仓管理+卖出信号
  positions.json    ← 持仓数据（gitignore）

notify/             ← 通知层
  wecom_sender.py   ← 企业微信发送器
  message_builder.py ← 消息模板
  config.py         ← Webhook配置

sector_screener/    ← 评分层（核心，谨慎修改）
  scorers/          ← 26个评分因子
  engine.py         ← 评分引擎+build_context
  config.py         ← 权重/阈值
  main.py           ← CLI入口

data_collector/     ← 数据采集层
  fetchers/         ← 10个数据源
  main.py           ← 采集入口

daily_pipeline/     ← 盘中流水线
  score.py          ← 16因子盘中评分
  backtest.sh       ← 回测脚本

web_dashboard/      ← Reflex前端
  web_dashboard/web_dashboard.py

research_data/      ← 盘中评分+回测数据
data/               ← 原始采集数据
```

## 常用命令

```bash
source .venv/bin/activate

# 数据采集
python -m data_collector.main --date=20260624

# 评分
python -m sector_screener.main --date=20260624 --top=10

# 黑天鹅
python -m portfolio.black_swan --date=20260624

# 持仓管理
python -m portfolio.manager check --date=20260624
python -m portfolio.manager list

# Dashboard
cd web_dashboard && reflex run
```
